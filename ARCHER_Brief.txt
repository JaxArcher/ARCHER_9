1. Context & Mission
ARCHER (Advanced Responsive Computing Helper & Executive Resource) is a personal AI assistant built for a single user. It is not a product or a platform. It is a JARVIS-style system that combines always-on voice interaction, passive behavioral observation via webcam and microphone, proactive pattern-triggered interventions, and a suite of specialized agent personalities — all running primarily on local hardware with cloud services available via a toggle.

This is a clean rebuild. The previous codebase is being retired (it is in the ARCHER_v2 subfolder). You are not patching or extending existing files. You are building from scratch, informed by lessons from prior iterations and by OpenClaw's proven architectural patterns.



2. Reference Material
OpenClaw (Primary Architectural Reference)
OpenClaw is an open-source personal AI assistant runtime with 117k+ GitHub stars. It has solved many of the infrastructure problems ARCHER needs. You are not running OpenClaw — you are porting its proven patterns to Python. Study it carefully before writing any code.

Repository: https://github.com/openclaw/openclaw

The specific OpenClaw subsystems to study and port:
Gateway / control plane — session model, WebSocket event bus, channel routing logic, presence tracking
Doctor / diagnostics — health check patterns, config validation approach
Skills/tools framework — how tools are discovered, registered, and called by agents
Cron + webhooks — the scheduling and trigger architecture
Security model — DM pairing, sandbox design, secret detection approach
Browser automation integration — how CDP control is wired into the agent tool layer

What NOT to port from OpenClaw:
Anything TypeScript-specific. Rewrite all patterns in Python — do not create Node.js subprocesses or bridges.
The messaging channel integrations (WhatsApp, Telegram, etc.) — not needed at this stage.
The macOS/iOS/Android node apps — Windows-primary, desktop-only for now.
ClawdHub skills marketplace — ARCHER does not use a public registry.

4. Dependency Isolation Strategy
Dependency conflicts destroyed previous ARCHER iterations. MediaPipe, ChromaDB, DeepFace, and PyTorch all have conflicting requirements if run in the same environment. The rule is:


Lock all package versions in pyproject.toml before writing any feature code. Run pip check as a CI step. Any conflict found during development must be resolved by moving the conflicting package to Docker — not by loosening version pins.


5. LLM Configuration & Cloud/Local Toggle
Default Configuration

Toggle Architecture
Cloud/local mode is a single runtime setting, not a compile-time flag. It must be switchable mid-session without restart. Implement as a ToggleService that all components subscribe to. When the user says 'switch to local mode' or flips the GUI toggle, every component that uses an LLM, STT, or TTS endpoint updates its active backend within the current session.

Store the toggle state in the central config DB (SQLite), not in memory.
Each service (STT, TTS, LLM) has a cloud_backend and a local_backend property. The ToggleService writes the active mode; each service reads it before every request.
The GUI orb color subtly indicates current mode: slightly warmer in cloud mode, cooler in local mode.
Auto-fallback: if a cloud API call fails (timeout, rate limit, no internet), automatically fall back to local and notify the user once. Do not fail silently.


6. Voice Pipeline
This is the most latency-sensitive part of the system. Every component must be benchmarked individually before integration.


Conversational Fillers
If the agent call has not returned a first token within 600ms, play an audio filler ('Let me think about that...', 'One moment...', 'Hmm...'). These are pre-recorded or pre-generated clips, not synthesized in real time. They prevent dead air and make the interaction feel natural.


7. Agent Architecture
Orchestration
Use LangGraph as the orchestration framework. The Orchestrator is a routing agent that classifies every inbound event (voice utterance, observer event, scheduled trigger) and dispatches it to the appropriate specialist agent. Agents do not call each other directly — all inter-agent communication goes through the Orchestrator.

Day 1 Agents

Phase 2 Agents (Architecture Must Support, Not Build Yet)
Knowledge / Second Brain — document ingestion, personal knowledge base, semantic search
R&D — system self-monitoring, capability research, model improvement proposals
QA — ethics and quality auditing of other agents' outputs

Agent Instruction Files
Each agent has a SOUL.md file in its workspace directory that defines its personality, boundaries, behavioral rules, and example interventions. The Orchestrator has an AGENTS.md that describes routing logic. These files are the primary way to tune agent behavior — code changes should be rare after initial build.

SOUL.md files are injected as the system prompt on every agent call.
They are readable and editable by the user without code changes.
Include explicit sections: Who You Are, How You Speak, When You Intervene, What You Never Do, Example Exchanges.


8. Memory Architecture
ARCHER uses a four-tier memory system. All tiers must be queryable by all agents. The Orchestrator pre-loads relevant context before dispatching to a specialist agent.


Inventory & Correction Memory (Assistant Agent)
The Assistant agent maintains a dedicated inventory table in the SQLite Tier 2 store. This tracks physical items the user owns — location, condition, last seen, category. The Assistant proactively uses this when the user asks about something they own, mentions looking for something, or makes a statement that contradicts a known inventory fact (e.g., 'I need to buy a new charger' when the inventory shows two chargers in the home office).

Schema: item_name, category, location, last_confirmed, notes, confidence_score
Items are added via voice ('ARCHER, note that my passport is in the top drawer of my desk')
Items are updated automatically when the user mentions their location in conversation
Proactive correction: if the user misstates something ARCHER knows (wrong location, wrong fact), the agent gently corrects it inline, not as a separate interruption


9. Observer Architecture
The Observer is ARCHER's ambient sensing layer. It runs continuously and silently. It never speaks to the user directly — it emits structured ObservationEvents that other agents consume.

What It Watches
Webcam: presence detection, pose estimation (sitting/standing/lying), emotion inference (DeepFace), food detection when food appears in frame
Microphone: VAD-based speech activity (seconds speaking vs silence, not content), background audio classification
System: active window title, application focus, keyboard/mouse activity level (activity score, not keylogging)

ObservationEvent Schema
Every event emitted by the Observer must include: timestamp, source (webcam|mic|system), event_type, confidence (0.0-1.0), evidence_pointer (frame_id or log_offset), and payload (structured dict). Agents must not act on events with confidence below 0.4 without corroboration.

Privacy Rules — Non-Negotiable
Raw video and audio are NEVER stored. Only derived features (events, labels, embeddings) are persisted.
A hardware-accessible kill switch in the GUI disables all observer inputs immediately.
The GUI always shows a visible indicator when the observer is active.
Raw AV capture for debugging is opt-in only, time-limited, and stored encrypted.

Isolation
MediaPipe (pose) and DeepFace (emotion) run in Docker containers due to protobuf conflicts with the main environment. The Observer sends frames to these containers via HTTP and receives structured results. Frame rate to containers is capped at 2fps to manage CPU load — the i9-10900K handles this path, not the GPU.


10. Desktop GUI
PyQt6 on Windows, native. This is a clean-slate implementation — do not reference or port the old orb code. The concept is preserved; the implementation is fresh.

Layout — Four Quadrants

GUI Rules
PyQt6 owns the main thread. No blocking calls anywhere in the GUI thread.
All updates to the GUI from background threads go through Qt signals/slots — never direct widget manipulation from non-GUI threads.
The webcam feed is optional — a small live thumbnail in the bottom-left when observer is active, hidden otherwise.
Cloud/local toggle is a visible toggle switch in the title bar, always accessible.
ARCHER HALT button is always visible and always functional. It cancels all queued actions and active TTS immediately.


11. PC Control & Browser Automation
The Assistant agent can control the desktop and browser on the user's behalf. All PC control actions require an intent confirmation step (ARCHER states what it is about to do and waits for verbal confirmation) before execution, except for read-only actions like taking screenshots.

Screen Control Stack
pyautogui — mouse, keyboard, window management
Playwright for Python — browser automation (not Selenium). Manages a dedicated Chromium instance.
pygetwindow — window enumeration and focus management
mss or PIL — screen capture for vision tasks ('what's on my screen right now?')

HALT Failsafe
At all times, the phrase 'ARCHER HALT' (or pressing the GUI button) immediately: stops all TTS playback, cancels all queued pyautogui actions, closes any active Playwright automation, and returns ARCHER to idle listening state. This is not optional and must be wired in before any PC control feature is enabled.


12. Scheduling & Proactive Intelligence
Cron / Scheduled Tasks
Port OpenClaw's cron architecture to Python using APScheduler (AsyncIO-native). All agents can register scheduled tasks. Tasks are stored in SQLite and survive restarts.

Nightly consolidation (2am): Tier 4 memory synthesis, agent self-evaluation reports
Market data refresh (configurable interval): Investment agent polling
Reminder delivery: time-based reminders set by user via voice
Weekly agent team review (Sunday evening if user initiates)

Pattern-Triggered Interventions
This is ARCHER's core differentiator from OpenClaw. Beyond scheduled cron, agents receive ObservationEvents from the Observer and decide whether to intervene based on pattern rules in their SOUL.md and pattern history in Tier 2 memory.

Intervention cooldowns are enforced per agent — no agent can intervene more than once per 2 hours on the same topic unless escalating a crisis.
The Orchestrator mediates conflicts — if Trainer and Therapist both want to intervene in a 10-minute window, Orchestrator decides which gets priority based on urgency scoring.
All triggered interventions are logged with their evidence chain (which ObservationEvents caused the trigger) for auditability.

14. Explicit Constraints — What Not To Do
These are lessons from previous build attempts. Treat them as hard rules.

Do not start with the GUI. Build the voice loop headless first.
Do not add a new package to the native venv without first checking it against the existing lock file. If it conflicts, it goes in Docker.
Do not put blocking calls in the PyQt6 main thread. Use Qt signals for all cross-thread communication.
Do not implement more than one agent at a time. Build Assistant, stabilize, then add Trainer, stabilize, then add the rest.
Do not implement Tier 3 or Tier 4 memory until Tier 1 and Tier 2 are working across session restarts.
Do not implement the Observer until the agent layer is stable. Observer events causing premature agent calls before agents are reliable will produce confusing bugs.
Do not change the TTS engine mid-build. Pick one (ElevenLabs for cloud, IndexTTS2 for local) and stick with it through Phase 4.
Do not try to run MediaPipe natively alongside the main Python process. It will conflict. Docker only.
Do not implement the voice cloning feature in this phase. It is Phase 2+.
Do not implement the Knowledge/Second Brain agent in this phase.
Do not implement the verifier model in this phase.

A. The Artifact Pane (Canvas)
The bottom-right quadrant of the GUI is the Artifact Pane — ARCHER's equivalent of this chat interface's right-side panel, where rich content is rendered alongside the conversation. This is not a static display. It is a dynamic, context-aware rendering surface that agents push content to whenever a visual output would be more useful than a spoken or text response alone.

The Artifact Pane replaces the need for ARCHER to read out data that is better seen — charts, tables, lists, maps, code, documents. When Investment gives you a portfolio summary, you see the graph. When Finance shows your budget, you see the chart. When Assistant finds a file, you see its contents. When Trainer tracks your session, you see the metrics panel.

What the Pane Renders

How Agents Push Content to the Pane
The artifact pane is driven by a standardized ArtifactPayload that any agent can emit as part of its response. This is separate from the spoken/text response. An agent can speak a summary and simultaneously push a detailed visual to the pane.

ArtifactPayload schema: { type: 'chart'|'table'|'document'|'code'|'image'|'web'|'dashboard'|'checklist', title: str, content: any, agent: str, timestamp: datetime }
The agent layer sends ArtifactPayload objects to a dedicated artifact queue. The GUI artifact thread picks them up and renders them — no blocking the main thread.
Each new artifact pushes a tab into the pane's tab bar. Up to 5 recent artifacts are kept as tabs. Older ones are accessible via an artifact history button.
The active tab auto-switches to the newest artifact when it arrives, unless the user has manually selected a previous tab — in that case it does not auto-switch.

Artifact Pane Tab Bar
The pane has a thin tab bar at the top showing the last 5 artifacts with agent color coding and a short label. Example tabs: '📊 Portfolio' (orange, Investment), '💰 Budget' (yellow, Finance), '🏃 Session' (green, Trainer). Clicking a tab restores that artifact. A history icon opens a list of all artifacts from the current session.



B. Orb Color Coding — Complete Specification
The main brief described the orb's state animations but omitted the per-agent color coding. This is a core part of ARCHER's personality — the orb's color tells the user at a glance which agent is active, even before a word is spoken.




C. Conversation Panel
The main brief described the top-left quadrant as 'Voice pipeline status: wake word indicator, VAD level meter, STT live transcription.' This is correct but incomplete — the conversation panel needs to persist the full session history, not just the live transcription of the current utterance.

Panel Layout
Top strip (thin): wake word indicator dot (lights up on detection) + VAD level meter bar + current STT transcription streaming in real time
Main area (scrollable): full conversation history for the current session
Bottom: cloud/local mode toggle + text input field (see section E)

Conversation History Entry Format
Each entry in the scrollable history shows:
User turn: timestamp, 'You:' label in neutral white, transcribed text
Agent turn: timestamp, agent name badge (colored to match orb color for that agent), ARCHER's response text
Observer events (optional, toggle): gray italic entries showing what the Observer detected — useful for debugging, off by default
System events: inline gray labels for mode changes, HALT events, session start/end



D. Voice Authentication
Voice authentication was present in all previous ARCHER specifications but was not included in the main brief. It should be included in Phase 1 — it is a voice pipeline component, not a later-phase feature, and it is simple to add at the same time as the STT pipeline.

Implementation
Library: SpeechBrain ECAPA-TDNN speaker verification model
Enrollment: On first run, ARCHER asks Col to speak 3 short enrollment phrases. These are stored as a voice embedding in the local SQLite DB.
Verification: After wake word detection and before the agent call, a 2-second voice sample is compared against the enrollment embedding. Cosine similarity threshold: 0.85.
If verification fails: ARCHER responds in a neutral tone, does not identify which agent would have responded, and does not execute any tool calls. Guest mode only.
Guest mode: A limited set of read-only interactions are permitted without authentication (ask the time, ask the weather, ask what ARCHER is). No memory reads, no tool execution, no personal data.



E. Text Input Fallback
ARCHER is voice-first but must support full text input as a fallback. This is both a usability feature (quiet environments, preference) and a development tool (faster testing without speaking).

A text input field sits at the bottom of the conversation panel, always visible.
Typing and pressing Enter (or clicking Send) routes the text through the same agent pipeline as a voice utterance — there is no separate text-mode agent path.
TTS is triggered by text input the same as voice input — ARCHER speaks its response unless TTS is muted.
A mute-TTS toggle sits next to the text input for when the user wants text-only mode.
The wake word is not required for text input — the text field is always active when the window is focused.


F. System Tray & Background Operation
ARCHER is an always-on system. Closing the main window should not stop ARCHER — it should minimize to the system tray and continue running the voice pipeline, observer, and scheduled tasks in the background.

Tray Icon Behavior
Tray icon shows ARCHER's logo. Color ring around the icon reflects current orb state (idle gray, listening blue, etc.).
Left-click the tray icon: restore the main window.
Right-click the tray icon: context menu with — Show Window, Mute Microphone, Pause Observer, Cloud/Local Toggle, Quit ARCHER.
'Quit ARCHER' in the tray menu is the only way to fully stop the process. The window's X button always minimizes to tray, never quits.
A tray notification (balloon tooltip) appears when ARCHER has a proactive message while the window is minimized — clicking it opens the window and focuses the conversation panel.



G. HALT — Voice Command Integration
The main brief described ARCHER HALT in the PC Control section, but it needs to be specified as a voice pipeline feature, not just a GUI button. The HALT command must be the highest-priority element in the voice processing loop.

How HALT Works in the Voice Pipeline
HALT detection runs as a parallel listener alongside the normal VAD/STT loop — it does not wait for a wake word.
When the phrase 'ARCHER HALT' is detected (or a configurable alternative), the following happens within 150ms: all queued TTS is stopped, all queued pyautogui actions are cancelled, all active Playwright sessions are closed, all pending agent calls are dropped, the orb turns red for 1 second then returns to idle.
ARCHER confirms verbally: 'Stopped.' — one word, then silence.
HALT does not clear conversation history or memory. It only stops active execution.
After a HALT, ARCHER is immediately ready for a new wake word — it does not require a restart.

Critical Threading Constraint
PyQt6 owns the main thread and its own event loop.
FastAPI runs in a dedicated thread with its own asyncio loop.
GPU inference (STT, TTS, LLM if local) runs in a worker thread pool behind a queue.
The webcam capture loop runs in its own thread.
These four components NEVER share objects directly. All inter-thread communication goes through thread-safe queues.
Violating this constraint is the single most common cause of GUI freezes and dropped audio frames. It must be designed in from Day 1.	
Isolation Rules
NATIVE (single venv, pyproject.toml, locked versions): FastAPI core, PyQt6, Faster-Whisper, openWakeWord, LangGraph, SQLite, Anthropic SDK, OpenCV. These must share one environment and must be tested together before any other package is added.

Agents should use the pane when it genuinely adds value, not for every response. A one-line answer does not need an artifact.

Implementation Note
Color transitions between agents should be smooth cross-fades (300ms), not instant snaps.
The orb never shows two colors simultaneously — the active agent always owns the orb fully.
If Observer fires a proactive trigger that activates Therapist, the orb transitions from gray to purple as Therapist begins speaking.	
Why This Matters
Without a persistent conversation panel, the user has no way to review what was said — especially important when ARCHER proactively intervenes and the user wants to read back what triggered it.
Agent badges in the conversation history also teach the user which agent handles what, naturally.	
Phase 1 Scope for Voice Auth
Enroll Col's voice on first run. Verify before every agent call. Reject unknowns into guest mode.
Do NOT implement multi-user profiles in Phase 1. The architecture should support it (separate embeddings per user_id in the DB) but only Col's profile is active.
Voice cloning attacks: not in scope for Phase 1. Note it as a Phase 3+ concern.	
Windows-Specific Note
Use QSystemTrayIcon (PyQt6 built-in). No third-party library needed.
On Windows, the tray icon area may hide inactive icons. Instruct the user during first-run setup to pin ARCHER to the visible tray area.	
Why HALT Is a Voice Pipeline Feature, Not Just a Button
If ARCHER is autonomously controlling the screen and the user's hands are not on the keyboard, the voice HALT is their only reliable escape.
The HALT listener must be active even when the main STT is busy processing another utterance.
This requires a second, lightweight, always-running speech detection thread dedicated solely to the HALT phrase.
